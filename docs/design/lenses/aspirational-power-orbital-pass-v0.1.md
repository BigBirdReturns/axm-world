# Aspirational Power Design Engine orbital pass v0.1

## Strongest direction

Make **the custody receipt the product’s proof spine**. RoDo(h) should not ask creators or players to believe a platform-scale future. It should let them perform one bounded act—insert, enter, walk, choose, resolve, return, retain—and then inspect the portable object that proves what happened. The runtime, custody object, acceptance test, creator documentation, and release page should each carry a distinct part of that proof.

This is an audit of a creator platform, playable runtime, custody object, and public product proposition. Capability truth: AXM-ARC authors validated cartridges; AXM-WORLD embodies them; the deterministic engine resolves choices; serialization retains ledger-backed consequence in a creator-owned run. Large creator economies, multiplayer worlds, and indefinite compatibility remain roadmap claims.

## Orbit findings

### Capability truth is unusually strong

`Observed:` proximity authority is separate from selection; resolution remains in the deterministic engine; cleared state and ledger facts derive transformed locations; cartridge export serializes the result; the full acceptance test reproduces the loop.

`Inference:` this clears the generic-futurism and evidence-collapse gates. The platform already has a stronger proposition than “a spatial renderer”: a portable authored world whose state can be independently inspected.

### The proof is distributed but not yet composed

`Observed:` runtime, test, cartridge panel, custody object, and repository documentation each expose pieces of the claim. No single proof map tells a creator or reviewer which artifact establishes which capability.

`Inference:` expert maintainers can reconstruct the story, but a public audience can still mistake visual polish for proof or overlook the custody object entirely.

`Proposal:` add a release-facing capability ledger with four columns: demonstrated capability, artifact/interaction, deterministic receipt, and explicit non-claim. The custody receipt—not a hero render—anchors the sequence.

### Compatibility and preservation costs are underrepresented

`Observed:` the cartridge panel shows engine version and identity; export uses a versioned run format. The product does not yet explain migration responsibility, supported compatibility windows, or what “permanent” means when engines change.

`Inference:` creator ownership is credible today, while indefinite preservation is not yet demonstrated. Calling world change permanent without this boundary transfers risk to creators and players invisibly.

`Proposal:` define “durable” as inspectable, exportable, reloadable under a declared compatibility contract. Publish a migration policy before claiming permanence across versions. Surface compatibility status on the cartridge receipt.

### The comparison language must remain internal

`Observed:` “Nintendo + Roblox” successfully names two constraints: tactile readability and extensible cartridges.

`Inference:` used publicly as a visual or market identity, it would weaken independent expression and encourage audiences to infer polish, scale, social systems, or trade dress that are not demonstrated.

`Proposal:` public language should lead with “creator-owned worlds that remember,” followed immediately by the seven-step proof. Nintendo and Roblox remain internal mechanism shorthand only.

## Product gate

The proof system is complete only when:

1. Every public capability claim links to a visible interaction or serialized field and a deterministic test.
2. Every roadmap claim is visibly labeled as future work.
3. The cartridge receipt names identity, engine compatibility, run format, ledger count, transformed locations, and custody/export status.
4. Creator documentation states ARC, WORLD, engine, theme, and cartridge responsibilities in one normative contract.
5. Compatibility and migration responsibility are explicit enough that “durable” does not imply unsupported permanence.
6. A fresh creator and a fresh player can each state what they own, what the platform guarantees, and what remains their risk.

Control question: **After one unassisted loop, can a creator or player prove what changed, who owns that state, and what compatibility guarantee applies without relying on RoDo(h)’s promotional language?**
