# Engine-design proposal — resource-spend as a bounded variance lever

**Status:** engine-design proposal. **No implementation.** This is the arc-first
engine-design pass called for by `RESOURCE_SPEND_DECISION.md` §"Sequence from
here" step 2. It turns the accepted invariant into a concrete, code-grounded
resolver design: the authored parameter shape, the exact threading into
`resolveChallenge`, the variance math, the gate/difficulty composition proofs,
the required property tests, and the receipt language. It is governed by
`POSITIONING.md` ("The anti-extractive contract") and `WORLDS_ROADMAP.md` §0a
(the encounter-agency taxonomy). **Resolver code may be written only after this
proposal is accepted, as its own reviewed engine change** (see "Acceptance
gate").

Read first: `RESOURCE_SPEND_MEMO.md` (the gate) and `RESOURCE_SPEND_DECISION.md`
(the invariant this proposal implements). The invariant, verbatim:

> Resource spend may reduce variance or protect against downside on an
> already-plausible attempt. It may not create success for an unqualified party,
> bypass role requirements, bypass authored difficulty, rescue a failed result
> after the fact, or become purchasable power.

"Already-plausible" is defined, not vibes: a party that meets the contract's
count and role gates — the exact booleans `evaluateParty` already computes as
`countOk` and `rolesOk` (`src/world/readiness.ts:305`, `:312`).

---

## 0. The design in one paragraph

The only permitted effect of `tokensSpent` is a **bounded, mean-preserving
narrowing of the symmetric (mean-zero) variance term of the roll.** It scales the
`variance = rng.uniform(-4, 4)` term in `scoreAgent` (`src/engine/resolver.ts:130`)
— and the symmetric `macroVariance` term for team checks
(`src/engine/resolver.ts:279`) — by an authored steadiness factor `k ∈
[minSteadiness, 1]`. Because those terms are symmetric with expectation `0`,
scaling them by `k` leaves the **expected score exactly unchanged** and strictly
**reduces the spread**. It touches nothing else: not the deterministic score, not
the thresholds, not the asymmetric character-owned `volatilitySwing`. A party that
fails on the mean still fails on the mean; a party already over the line just
stops losing to bad luck. The lever is authored per contract (never global),
refused upstream when the gates fail, and a no-op inside the resolver for any
below-gate party that reaches it anyway.

---

## 1. Authored parameter shape (where a cartridge declares the lever)

The lever is **authored, never global.** A new optional field is added to the
`Challenge` record (`src/engine/types.ts:279`), with an optional per-check
override on `MechanicCheck` (`src/engine/types.ts:218`). Optionality is the
sovereignty rule in the schema: a contract that authors nothing surfaces nothing.

Proposed additions to `src/engine/types.ts`:

```ts
/** An authored, per-contract resource-spend lever. When present, the encounter
 *  may spend the arc's token (arc.tokenName) to steady an already-plausible
 *  attempt. Its only effect is a bounded, mean-preserving narrowing of the
 *  roll's symmetric variance — never a score, threshold, or outcome change.
 *  Absent = the contract authors no spend lever, and no spend option renders. */
export interface ResourceSpendLever {
  /** Max tokens this contract honors toward steadiness. Spend beyond this is
   *  refused upstream and clamped in-resolver; it is not "more power for more
   *  tokens" past the cap. Authored, per contract. */
  maxTokens: number;
  /** Fraction of the symmetric variance band removed per honored token.
   *  0..1. e.g. 0.2 = each token shaves 20% off the ±band. */
  steadinessPerToken: number;
  /** Hard floor on the steadiness factor k. (0,1]. The band never narrows below
   *  this, so a marginal party always keeps a real failure chance (see §5, §7c).
   *  e.g. 0.5 = the roll can be steadied to at most half its natural spread. */
  minSteadiness: number;
}

// on MechanicCheck (optional per-check override; else the challenge value applies)
resourceSpend?: ResourceSpendLever;

// on Challenge (the encounter-level lever)
resourceSpend?: ResourceSpendLever;
```

Notes:

- **Reuses the existing token**, not a new currency stack: the arc's authored
  token (`Arc.tokenName`, regenerated by `Arc.tokensPerCycle`, capped at
  `Arc.maxTokens` — `src/engine/types.ts:442`–`:443`). This satisfies the memo's
  Q1 ("no confusing currency stacks") and needs no new economy plumbing —
  `tokensSpent` already lives on `ChallengeAssignment` (`src/engine/cycle.ts:35`)
  and is already debited by `spendTokens` (`src/engine/cycle.ts:177`).
- **Not scaled by difficulty modes.** `applyDifficultyMode` scales
  `difficultyThreshold`, `difficultyRating`, `timePressure.aggregateThreshold`,
  and reward multipliers (`src/engine/difficulty.ts:37`–`:63`) — it must **not**
  touch `resourceSpend`. Like drop rates, the spend lever's magnitude is the
  designer's authored call, not a blanket multiplier (see §6).
- **Non-purchasable is a standing rule, not a field.** Nothing here sells tokens;
  they are earned via `tokensPerCycle` only. The schema cannot express a purchase.

## 2. Where `tokensSpent` enters `resolveChallenge`

**The current fact.** `ChallengeAssignment.tokensSpent` is debited by `runCycle`
at `src/engine/cycle.ts:175`–`:182` (`spendTokens(org, assignment.tokensSpent)`),
but the call to `resolveChallenge` at `src/engine/cycle.ts:193`–`:200` never
passes it. So today the token spend is a pure ledger cost with zero effect on the
roll — which is exactly why surfacing it as agency today would be fake agency
(`WORLDS_ROADMAP.md` §0a, resource-spend row).

**The threading (three edits, all in engine — deferred to Acceptance gate):**

1. **`ResolveChallengeOpts`** (`src/engine/resolver.ts:15`–`:22`) gains one field:

   ```ts
   /** Tokens the assignment committed toward steadiness. 0 = no spend.
    *  Honored only when the party clears the same count+role gates the shell
    *  used to offer the option; a no-op otherwise (see §5). */
   tokensSpent: number;
   ```

2. **The call site** (`src/engine/cycle.ts:193`) passes it through:

   ```ts
   const report = resolveChallenge({
     challenge: effectiveChallenge,
     assignedAgents: agentList,
     org, arc,
     rng: step1Rng.fork(),
     cycle,
     tokensSpent: assignment.tokensSpent,   // ← new
   });
   ```

   No new spend happens here; `spendTokens` at `:177` already performed the debit.
   This only *informs* the roll of what was already spent.

3. **`resolveChallenge`** (`src/engine/resolver.ts:236`) computes a single
   `steadiness` factor `k` from the authored `resourceSpend` lever, the honored
   `tokensSpent`, and the in-resolver gate check (§5), then threads `k` into
   `scoreAgent` (`src/engine/resolver.ts:113`) and the team-aggregate
   `macroVariance` (`src/engine/resolver.ts:279`). `scoreAgent` gains a
   `steadiness: number` parameter defaulting to `1`.

`k` is a pure function of authored parameters + `tokensSpent`; it reads no
wall-clock and no session state, so determinism is preserved (§7d).

## 3. What variable it affects (variance, explicitly not the mean)

The resolver's per-agent score (`scoreAgent`, `src/engine/resolver.ts:139`–`:141`):

```
S = rawScore + gearBonus + relMod + moraleMod + afflictionMod + V + W + traitBonus
```

Its two random sources:

- **`V = rng.uniform(-4, 4)`** — `src/engine/resolver.ts:130`. **Symmetric,
  `E[V] = 0`**, band `±4`, `Var[V] = 8²/12 ≈ 5.33`. This is ordinary luck-of-the-roll.
- **`W = getVolatilitySwing(...)`** — `src/engine/resolver.ts:133`–`:135`, defined
  at `:68`–`:72`. **Asymmetric**: `uniform(-5, 10)` (mean `+2.5`) or `uniform(-3, 5)`
  (mean `+1`) or `0`, driven by the agent's hidden `volatility` (or forced to max
  by the `Reckless` affliction). Its mean is **not** zero and it encodes authored
  character temperament.

For team-aggregate checks there is one more symmetric source:

- **`macroVariance = rng.uniform(-3, 3) * (partySize / 2)`** —
  `src/engine/resolver.ts:279`. **Symmetric, mean 0.**

**The lever affects the symmetric, mean-zero terms only — `V` and
`macroVariance`.** It multiplies them by `k`. It deliberately leaves `W`
(`volatilitySwing`) untouched, for two reasons: (a) `W` is asymmetric, so scaling
it would move the mean and violate clause 2 of the invariant ("No mean lift past
entitlement"); (b) `W` is character-owned risk (temperament, the `Reckless`
affliction), which is authored identity, not luck the player should be able to buy
away. The first shape steadies *the roll's ordinary luck*, not *who the agent is*.

This is what makes "reduces variance, not the mean" a **theorem, not a promise**
(§4): the only terms the lever can reach have expectation `0`.

## 4. How the narrowing works (the math sketch)

Let `k = steadiness ∈ [minSteadiness, 1]`, with `k = 1` meaning "no spend." Define
the honored token count and factor:

```
honored = min(tokensSpent, resourceSpend.maxTokens)          // cap enforced here
k       = max(minSteadiness, 1 - steadinessPerToken * honored) // floor enforced here
```

`k` is a pure, deterministic function of authored parameters and `tokensSpent`.

The resolver replaces the two symmetric terms with their scaled forms:

```
V'           = k * rng.uniform(-4, 4)          // per-agent (resolver.ts:130)
macroVariance' = k * rng.uniform(-3, 3) * (partySize / 2)   // team (resolver.ts:279)
```

**Mean invariance (proof).** `E[V'] = k · E[uniform(-4,4)] = k · 0 = 0 = E[V]`. The
deterministic part of `S` is unchanged and `W` is untouched, so `E[S'] = E[S]`
**exactly**, for every `k`. The expected score — the party's entitlement — does
not move. A party that fails on the mean still fails on the mean.

**Strict spread reduction (proof).** `Var[V'] = k² · Var[V]`. For any `k < 1`,
`Var[V'] < Var[V]`; likewise for `macroVariance'`. All other terms of `S` are
unchanged, so `Var[S'] = k²·Var[V] (+ k²·Var[macro]) + (unchanged terms) ≤
Var[S]`, strictly `<` whenever `k < 1` and the party carries any symmetric
variance. The roll gets tighter around the same mean.

**Band geometry.** `V'` ranges over `[-4k, +4k] ⊆ [-4, 4]`. Narrowing raises the
worst case by `4(1-k)` and lowers the best case by `4(1-k)` — symmetrically. Fewer
unlucky losses **and** fewer lucky wins. A party comfortably over the threshold
loses fewer runs to bad luck (the honest benefit); a party under the threshold
wins fewer runs to good luck (correctly — spend does not rescue it); a party
exactly at the threshold stays ~50/50 (symmetry preserves it).

**Bounded, so failure stays reachable.** Because `k ≥ minSteadiness > 0`, the band
never collapses to a point. A marginal party (mean at or just over the line)
always retains a real, nonzero chance of failure — spend cannot buy certainty
(§7c). This is the single most important guardrail against "spend-to-guarantee."

**Player-facing projection.** `readiness.ts` already models the resolver's
variance as the `SWING = 5` buffer a check must clear to read "reliable"
(`src/world/readiness.ts:13`, `classify` at `:234`). Spend visibly tightens that
buffer in the projection: a "thin" check (`margin` within `±SWING`) narrows toward
"ready" without its `projected` score (the mean) moving. The projection shows the
band shrinking around a fixed center — never the center sliding up — before the
player commits (decision memo: "Visible before resolve").

## 5. Why it cannot bypass `countOk` / `rolesOk` (gate independence)

Two independent enforcement layers, so the guarantee holds even if one is bypassed
(replay, direct API, malformed assignment):

**Layer 1 — refused upstream (primary).** The encounter shell offers the spend
option only when the party clears the gates `evaluateParty` computes:
`countOk` (`src/world/readiness.ts:305`: party length within `[minAgents,
maxAgents]`) and `rolesOk` (`src/world/readiness.ts:312`: every required role
covered). When `!countOk || !rolesOk` the option does not render and the
assignment's `tokensSpent` stays `0` — exactly the way the difficulty-mode chooser
is withheld when a cartridge authors no modes (`POSITIONING.md`,
"Difficulty-mode contracts render posture choice"). This mirrors the existing
lock path in `runCycle`, where an inaccessible challenge "never resolves and never
spends tokens" (`src/engine/cycle.ts:143`–`:158`).

**Layer 2 — no-op in the resolver (defensive).** `resolveChallenge` recomputes the
same gates from data it already holds — `challenge.rosterRequirements`
(`minAgents`, `maxAgents`, `roleRequirements`) against `assignedAgents` — and
forces `k = 1` (no effect) whenever the party is below min/over max or missing a
required role. So a below-gate party resolves **bit-identically** with or without
`tokensSpent`. The steadiness modifier is a strict no-op outside the authored risk
envelope.

Gate independence is therefore total: spend has **zero** influence on `countOk` /
`rolesOk` (those booleans never read `tokensSpent`), and `tokensSpent` has zero
influence on a below-gate party's roll. This is invariant clause 1
(`RESOURCE_SPEND_DECISION.md` §"engine invariant").

## 6. Why it cannot bypass authored difficulty

Difficulty modes are a pure challenge transform applied **before** the resolver:
`runCycle` calls `applyDifficultyMode(challenge, mode)` at
`src/engine/cycle.ts:171`, and the resolver stays mode-blind
(`src/engine/difficulty.ts:1`–`:6`). `applyDifficultyMode` raises
`difficultyThreshold` on every check by `difficultyMultiplier`
(`src/engine/difficulty.ts:37`–`:40`) and scales `difficultyRating` and time
pressure similarly.

The steadiness lever is **orthogonal** to this transform:

- It operates on the roll's **variance**, never on the **threshold**. A Heroic
  mode raises the bar; spend narrows the roll around the *same, unchanged* mean.
- `applyDifficultyMode` never reads `resourceSpend`; the steadiness factor never
  reads any threshold. Neither can leak into the other.
- Composition order is safe by construction: difficulty transforms the challenge
  (harder threshold) → resolver applies steadiness to the mean-zero variance. Since
  `E[S'] = E[S]` (§4), a party that cannot clear the mode-scaled threshold **on its
  mean** still cannot clear it — spend removes luck, not distance. Spend can never
  lift a party over a Heroic bar it has not earned on the mean.
- `resourceSpend` lives on the base `Challenge`/`MechanicCheck` and is **not**
  multiplied by `difficultyMultiplier` (§1). A mode's `mechanicAdditions`
  (`src/engine/types.ts:425`) may author their own `resourceSpend`, unscaled — the
  mode designer's explicit call, consistent with how `applyDifficultyMode` treats
  additions as "written for the mode, not multiplied by it"
  (`src/engine/difficulty.ts:30`–`:31`).

This is invariant clause 2 (no mean lift past entitlement) composed with authored
difficulty: harder authored difficulty raises the mean the party must reach; spend
never raises the party's mean.

## 7. Required property tests (must pass before any UI ships)

All four are gates on the engine change (§Acceptance gate), asserted against the
real resolver with fixed seeds. They map 1:1 to the invariant clauses.

- **(a) A hopeless party's outcome distribution is identical with and without
  spend.** Take a party that fails a check by more than the whole variance band
  (mean + `4` < threshold) *and* a party that is below-gate (`!countOk` or
  `!rolesOk`). For a sweep of seeds and `tokensSpent ∈ {0..maxTokens}`, assert the
  `RunReport.outcome` (and the per-check `passed` vector) is identical to the
  `tokensSpent = 0` run. Two mechanisms guarantee this: the resolver no-op for
  below-gate parties (§5, Layer 2), and the bounded band never reaching the
  threshold for a truly hopeless above-gate party (§4). Enforces clause 1 +
  "cannot create success for an unqualified party."

- **(b) A plausible party's variance strictly narrows.** Take a party that clears
  the gates and sits above the threshold. Over N seeds, compute the empirical
  variance of the check `score` at `tokensSpent = 0` vs. `> 0`. Assert
  `Var(spend) < Var(no-spend)` strictly, and that it decreases monotonically as
  honored tokens rise (until the `minSteadiness` floor). Enforces "reduce
  variance."

- **(c) The failure floor never rises (two-pronged).**
  - **(c1) Mean invariance.** Over N seeds, assert the empirical mean of the
    check `score` is statistically indistinguishable across `tokensSpent` values
    (and, for the deterministic components, exactly equal). The expected score —
    the "failure floor" that decides whether a party fails on the mean — does not
    move. Enforces clause 2 ("A party that fails on the mean still fails").
  - **(c2) Failure stays reachable.** For a marginal party (mean at/just over the
    threshold), assert `P(failure) > 0` at maximum spend — the bounded
    `minSteadiness` floor keeps a real loss chance; spend never manufactures
    certainty. Enforces "cannot become a guaranteed pass."

- **(d) Determinism preserved.** Assert same arc + seed + assignment (including
  `tokensSpent` and `difficultyModeId`) → byte-identical `RunReport`
  (`narrativeSeed`, per-agent `mechanicResults`, `outcome`, `lootDrops`). `k` is a
  pure function of authored params + `tokensSpent`; the roll still derives from
  `hashSeed(org.rngSeed, cycle, challenge.id)` (`src/engine/resolver.ts:240`).
  Enforces clause 3 and the platform's core determinism claim (`POSITIONING.md`).

Tests (a), (b), (c1) are the exact assertions `RESOURCE_SPEND_DECISION.md` §"engine
invariant" names as mandatory before any UI ships ("a hopeless party's outcome
distribution is identical … a plausible party's variance strictly narrows without
its failure floor rising").

## 8. Receipt language requirements

The receipt must state, in **player language**, what was spent and what it bought —
no engine notation. Required:

- **Spend line, plain language.** On spend: `"Spent 1 {token}: steadied the roll"`
  (pluralized per count, `{token}` = `arc.tokenName`). On hold: `"Held your
  {token}"`. Never surface `k`, `steadinessPerToken`, `variance ±4→±2.4`,
  probabilities, or any resolver term. This matches the receipt doctrine and the
  decision memo's "Receipt-recorded" clause.
- **Ledger shows the debit.** The token decrement already lands in the ledger via
  `spendTokens` (`src/engine/economy.ts`, called at `src/engine/cycle.ts:177`),
  which reduces `org.resources.tokens` (`src/engine/types.ts:186`–`:190`). The
  receipt's ledger section must show that debit as a real cost, so spend reads as a
  genuine sink, not free safety.
- **Pre-commit framing only.** The spend line describes a choice made *before*
  resolve (alongside the deploy step), never a post-outcome upsell. There is no
  "you failed — spend to retry" surface; the schema cannot express one.
- **Report plumbing (shell, not resolver).** The play-pipeline receipt
  (`summarizeReport` → `PlayReportView`, `WORLDS_ROADMAP.md` §0 projection chain)
  reads `tokensSpent` from the assignment and the debit from the ledger to render
  the line. If a machine-readable spend record on `RunReport` is wanted, add an
  optional `resourceSpend?: { tokensSpent: number; effect: "steadied" | "held" }`
  — but the *language* stays player-facing and lives in the shell/i18n layer, not
  the engine.

## Design constraints carried forward (open memo questions)

Two memo questions (`RESOURCE_SPEND_MEMO.md` Q4–Q5) touch the resolver's
neighbors and must be settled in the shell/economy review, not silently:

- **Refund interaction (memo Q4).** `Outcome.tokenRefund` exists
  (`src/engine/types.ts:277`). The first shape treats spend as a **real sink**: a
  pre-commit variance purchase that is consumed. Any refund-on-failure must not
  make spend net-positive in expectation (that would be grind padding / free
  safety, memo Q5). Refund policy is authored per outcome and is **out of scope for
  the resolver modifier**; flag it for the economy review so "spend, fail, get it
  back" cannot become a loophole.
- **Holding must stay viable (memo Q5).** Because the lever only trims symmetric
  variance and never lifts the mean, spend is never strictly required to pass a
  contract a party is entitled to — holding tokens is a legitimate strategy. This
  is a property the balance harness should watch, not a resolver assertion.

The asymmetric "absorb a bounded amount of downside variance" phrasing in the
decision memo is **explicitly deferred** past this first shape. A pure one-sided
downside clamp (e.g. `Math.max(V, -cushion)`) raises the mean and so violates
clause 2; the only mean-preserving "downside protection" is symmetric narrowing,
which is what this proposal ships. Any future asymmetric variant is a separate,
separately-reviewed engine change with its own mean-invariance proof.

## Acceptance gate

This document is a proposal. It authorizes no code.

1. **Accept this proposal** as the engine design for the resource-spend lever.
2. **Then, and only then**, write the resolver change as its **own reviewed engine
   change**: the `ResourceSpendLever` schema (§1), the `tokensSpent` threading into
   `resolveChallenge` (§2), the steadiness math (§4), the two-layer gate
   enforcement (§5), and the four property tests (§7) — which must be green before
   merge. No shell/UI work rides on this commit.
3. **Only after the resolver honors the lever and the property tests pass**, build
   the encounter-shell surface — the pre-commit spend option, the risk-band
   projection, and the receipt — **gated exactly like difficulty-mode choice**: a
   contract that authors no `resourceSpend` shows no spend option (as First Charter
   authors no mode and shows no chooser; Karazhan authors Heroic and shows one —
   `POSITIONING.md`, "Difficulty-mode contracts render posture choice"). The option
   renders only when `resourceSpend` is authored **and** `countOk && rolesOk` **and**
   the player holds tokens.

Only at that point does this become the fourth encounter-agency taxonomy row —
"resource-spend choice" — in `WORLDS_ROADMAP.md` §0a, promoted from **blocked**
to **implemented**. Until then the UI keeps refusing to imply the lever exists.

## Control question (must answer yes to all three)

> Can tokens become an honest risk-management lever while the engine still refuses
> hopeless parties, preserves authored difficulty, and records exactly what the
> player spent and why?

- **Refuses hopeless parties** — yes: gate independence (§5) + bounded band (§4),
  asserted by test (a)/(c2).
- **Preserves authored difficulty** — yes: mean invariance composed orthogonally
  with `applyDifficultyMode` (§6), asserted by test (c1).
- **Records what was spent and why** — yes: player-language receipt + ledger debit
  (§8).

All three hold simultaneously, so the proposal clears the gate.
