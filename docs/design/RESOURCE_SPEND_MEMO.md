# Design memo — resource-spend as an honest authored lever

**Status:** design memo. **No implementation.** This memo must be resolved before
any `tokensSpent`-in-resolver work begins.

**Why a memo first.** Resource-spend is the encounter-agency row (`WORLDS_ROADMAP`
§0a) that touches the *economy*, not just a surface projection. It is the side
door through which extraction grammar re-enters — spend to reduce risk, spend to
save time, spend to retry, spend to avoid loss. Each of those can be honest, but
only if cost, scarcity, recovery, failure modes, and receipt language are defined
*before* code. This memo is governed by `POSITIONING.md` ("The anti-extractive
contract").

## Control question

> Can resource-spend become an honest authored lever without becoming monetized
> impatience, fake safety, or a bypass around the cartridge's intended risk?

If the answer to any sub-question below lands on the extractive side and cannot be
designed out, resource-spend does **not** ship — the row stays future, and the UI
keeps refusing to imply the lever exists.

## The current fact (why it's not a lever yet)

Today `ChallengeAssignment.tokensSpent` is debited by `runCycle`
(`engine/cycle.ts` → `spendTokens`) but **never reaches `resolveChallenge`** — it
does not affect the roll, the thresholds, or the outcome. So spending a token is
strictly a cost with no effect. Surfacing it as a choice today would be fake
agency. Making it a real lever means the resolver must *honor* it — which is an
engine-semantics change, hence this memo.

## The questions the memo must answer (before code)

1. **What resource can be spent?** The cartridge's authored token
   (`arc.tokenName`, regenerated by `arc.tokensPerCycle`, capped at
   `arc.maxTokens`) — or a new authored resource? Reuse vs. new stack has
   anti-extraction consequences (no "confusing currency stacks").
2. **What engine variable does it affect?** Candidates, each with a different
   risk profile:
   - a variance/volatility reduction (narrower roll) — "fair safety";
   - a flat score bonus — closest to pay-to-win-shaped;
   - an extra attempt/retry — closest to "spend to avoid loss";
   - a reroll of a failed check — same.
   The choice must be *authored by the cartridge*, not global engine behavior, or
   it violates sovereignty (every arc would inherit it).
3. **What is the cost?** Fixed? Scaling with difficulty? Authored per contract?
4. **Is the cost recoverable?** Today failure already refunds via
   `outcomes.*.tokenRefund`. Does spend interact with refund? Can a player spend,
   fail, and get most of it back (low stakes) or is it a real sink?
5. **Can the player grind it?** `tokensPerCycle` regenerates tokens. If spend is
   always net-positive in expectation, it becomes mandatory optimization (grind
   padding). The design must make holding tokens a legitimate strategy.
6. **Can it create pay-to-win later?** Tokens are earned, not bought — but if a
   future monetization ever sells tokens, spend-to-win is one config change away.
   The memo must state the standing rule: **tokens are never purchasable**, and
   put that in `POSITIONING.md`'s Reject list explicitly if it isn't covered by
   "paid power."
7. **Can it bypass authored difficulty?** If spend can beat a check the roster
   could never pass, it bypasses the cartridge's intended risk (and would trivially
   defeat difficulty-mode choice). Spend must **shift odds within** the authored
   envelope, never guarantee a pass the party hasn't earned.
8. **Can it turn failure into a pricing surface?** "You failed — spend N to retry"
   is the canonical dark pattern. If retry/reroll is the chosen effect, it must be
   a *pre-commit* choice, never a post-failure upsell.
9. **How does the receipt explain the spend?** The result must say, in player
   language, what was spent and what it bought ("Spent 1 {token}: steadied the
   roll" / "held your {token}"), consistent with the receipt doctrine (no
   engine notation). The ledger must show the debit.
10. **When should the UI refuse to show the spend option?** When the cartridge
    doesn't author it; when the player has zero tokens; when the effect would push
    a hopeless party to a guaranteed pass (bypass); and any state where showing it
    would imply an unavailable lever.

## Provisional recommendation (to be challenged, not adopted)

If we do this at all, the *least* extractive shape is: an **authored, per-contract,
variance-reducing** spend — it makes a risky-but-earned party *more reliable*
without changing whether a hopeless party can pass, is a real token sink (holding
is viable), is a **pre-commit** choice (never a failure upsell), and is explained
plainly in the receipt. It becomes a fourth taxonomy row ("resource-spend choice")
only once the resolver honors it and every question above is answered on the
non-extractive side.

## Out of scope for this memo

Implementation, engine changes, and UI. This memo is the gate; the arc-first
engine design pass is the next artifact only if this memo clears the control
question.
